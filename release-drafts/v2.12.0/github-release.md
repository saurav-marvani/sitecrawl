# Firecrawl v2.12.0

## Improvements

- **URL-level threat protection** — Added an enterprise, flag-gated (`threatProtection: "disabled" | "allowed" | "forced"`) domain- and URL-risk classifier that enforces across **scrape, batch scrape, crawl, search, map, extract, and agent** on v1 and v2. In `normal` mode requests are classified against Google Web Risk's Update API using locally synced hash-prefix lists (MALWARE, SOCIAL_ENGINEERING, UNWANTED_SOFTWARE) and per-URL host-suffix × path-prefix expressions per the Safe Browsing spec, so a flagged page is caught even on an otherwise clean domain. Redirects are re-checked on any URL change, blocked responses raise `unsafe_domain_blocked` (403 on sync, carried through queued jobs and webhooks), and blocked links inside a crawl are silently skipped with a per-crawl `threat_blocked` counter. Local rules (`whitelist`, `blacklist`, `blockedTlds`, `riskScoreThreshold`, `failurePolicy`, `allowRequestOverrides`) resolve without a paid scan; verdicts are never persisted (ZDR) and only deduplicated in-memory per request/job. Billed at **+2 credits per unique scanned URL**; teams without the flag see zero added latency and no Redis or provider traffic.
- **IP allowlists on API keys** — Added a per-team API key IP allowlist gated by the enterprise `ipRestriction` team flag. When the flag is on and the allowlist is non-empty, API-key and OAuth (`fco_`) requests from any other IPv4/IPv6/CIDR entry (with IPv4-mapped-IPv6 normalization) are rejected with 403 linked back to the settings page. Allowlists live in the new `ip_restriction_config` table, are read outside the ACUC RPC with a 60 s Redis cache, fail closed on lookup errors, and empty/missing lists remain unrestricted so a team cannot lock itself out before configuring. New `POST /admin/<BULL_AUTH_KEY>/ip-restriction-cache-clear` invalidates a team's cache immediately from the dashboard.
- **Per-key scope and format lockdown** — Added the enterprise `keyRestriction` team flag, which lets a team lock individual API keys to an allowlist of **output formats** and/or **endpoint groups** (`scrape`, `crawl`, `batch-scrape`, `map`, `search`, `extract`, `agent`, `parse`, `browser`, `monitor`, `research`, …). Enforcement runs server-side with **no per-request override** — a locked key handed to an agent environment cannot be promptable into fetching raw HTML, using the `screenshot`/`scrape`/`executeJavascript`/`pdf` actions, or calling other endpoints. Config lives in `key_restriction_config` (one row per `api_key_id`, `allowed_formats` + `allowed_endpoints` jsonb) with a 60 s Redis cache; unknown paths deny by default (fail-closed for future endpoints), v1 alias formats normalize to v2 types, and v0 is denied outright on restricted keys.
- **Per-key usage limits enforced via Autumn** — API-key spend limits (the customer-facing "Set limit" flow in firecrawl-web) are now enforced on the server. Every `checkCredits` call site — the shared `checkCreditsMiddleware` covering scrape/crawl/batch/map/search/extract/agent/monitor/parse on v1 & v2, the v2 `browser` and `scrape-browser` controllers, `fireclaw`, and v0 scrape/crawl/search — now forwards `apiKeyId` in Autumn's check `properties` (mirroring the tracking property already sent). Autumn can now return 402 when an individual key has exceeded its configured daily/weekly/monthly limit, without touching credit balances for the team as a whole. Complements the new `ipRestriction` + `keyRestriction` per-key controls.
- **Session Replay for `/v2/interact`** — Added `GET /v2/interact/:sessionId/replay` (with a `/browser/...` alias) returning per-tab replay metadata with `pages`, `pageUrl`, and timings, and `GET /v2/interact/:sessionId/replay/:pageId` returning an HLS `.m3u8` playlist. Playlists are stitched from pre-signed segment URLs that expire (~6 h) — clients re-fetch the playlist to refresh. Recording is on by default; pass `"recordSession": false` on session create to opt out. `scrape-interact` responses now include `sessionId` so callers can link jobs to replays, and replay endpoints return 503 unless `BROWSER_SERVICE_URL` is set.
- **`search` monitor target** — Added a first-class `search` monitor target that runs SERP queries on a schedule and alerts on new or materially-developed results, alongside the existing `url` target. The target accepts `queries[]`, `searchWindow`, `includeDomains[]`, `excludeDomains[]` (exclude wins on overlap), and `maxResults`, with effective depth derived from `judgeEnabled` (`raw` returns SERP hits with no LLM calls when off, `deep` runs the LLM judge stack when on). Goal-versioned URL/event dedupe (keyed on goal + monitor name + queries), stable opaque event keys to prevent label-drift duplicates, cadence re-judging even when SERP snippets are unchanged, and a `material_dev` alert mode that re-alerts only on material developments of a known event. Exposed across the JS, Python, Java, .NET, Go, PHP, Ruby, Rust, and Elixir SDKs with a `MonitorSearchTargetResult` type.
- **Search monitor billing** — Search monitors now bill at a flat, deterministic rate per check: `SEARCH_CREDITS` is charged `ceil(totalResults / 10) * 2`, judged results are charged 1 credit per result (down from 5), and `raw` mode pays nothing for the judge. Deep-mode per-page scrapes flow through the shared scrape pipeline as queued jobs (billed under `CREDITS` like any other scrape) rather than being double-charged on the search check. Actuals are recorded on the check as `flatSearchTargetCredits`, and search-target pages carry 0 per-page credits. Estimates surfaced at monitor create/update include both the search call and a per-result judge allowance.
- **`menu` scrape format** — Added a new `menu` format that returns deterministic structured restaurant-menu data as `document.menu`, mirroring the existing `product` format end-to-end. Beta-gated via the `menuBeta` team flag (teams without the flag get no `menu` field and no error), with a `MENU_EXTRACTION_SERVICE_URL` self-host kill-switch: if the upstream menu engine is unreachable the scrape still succeeds with a `Menu format is not available` warning instead of failing. Exposed across the JS, Python, Java, .NET, Go, PHP, Ruby, Rust, and Elixir SDKs, with the Rust SDK bumped from `2.9.1` to `2.10.0` since `Format::Menu` is a source-breaking addition to its public `Format` enum.
- **Hosted `/v2/parse` file uploads** — Added an authenticated two-step upload flow for parsing local files without exposing a Firecrawl API key to the model or shell. `POST /v2/parse/upload-url` mints a short-lived, team-scoped `uploadRef` plus signed upload parameters; the client uploads bytes directly to storage; and `POST /v2/parse` then accepts JSON with `uploadRef` and reuses the existing parse pipeline. Refs are HMAC-signed with a 10-minute expiry and bound to team, object path, filename, content type, and max size; production storage uses GCS signed POST policies with `content-length-range` enforcement; uploaded objects are best-effort deleted after parse. Multipart `/v2/parse` continues to work unchanged.
- **Slack App** — Added a first-party Slack integration so teams can connect a workspace, receive monitor alerts in a Slack channel, and drive monitors from Slack via a `/monitor` slash command (`watch`, `list`, `cancel`, `status`, `help`, with a bare URL treated as `watch`). OAuth (`POST /v2/slack/oauth/start`, `GET /v2/slack/oauth/callback`) uses Redis-backed state with a TTL and redirect sanitization; management endpoints cover install status, channel listing, and uninstall (`GET /v2/slack/status`, `GET /v2/slack/channels`, `DELETE /v2/slack/installation`); event and command webhooks (`POST /v2/slack/events`, `POST /v2/slack/commands`) verify `X-Slack-Signature` against the captured raw body. Monitor alerts render as Block Kit summaries with strict Slack-limit guards, safe link rendering, auto-join of public channels on first post, and judge-based noise suppression that reads authoritative check counters instead of the truncated page list. Bot tokens are stored AES-256-GCM-encrypted in a new `slack_installations` table (with a `plain:` self-host fallback), and monitors gain a `notification.slack` config (`enabled`, `channelId`, `channelName`).
- **PHP SDK Laravel AI tools** — Added native tool classes for [laravel/ai](https://laravel.com/docs/13.x/ai-sdk) so PHP developers can drop Firecrawl into any Laravel agent's `tools()` array without hand-rolling HTTP: `FirecrawlScrape` (`firecrawl_scrape`), `FirecrawlSearch` (`firecrawl_search`), `FirecrawlMap` (`firecrawl_map`), `FirecrawlCrawl` (`firecrawl_crawl`), and a `FirecrawlTools::all()` helper. Tools implement `Laravel\Ai\Contracts\Tool`, resolve the existing `FirecrawlClient` singleton from the container (config and auth are reused), and treat `laravel/ai` as a suggested dev dependency so the SDK's PHP 8.1 floor is unchanged. Tool outputs are budgeted for model context — scrape truncates at 80k characters, crawl pages at 15k under a 100k whole-result budget, search/map drop tail items with an explicit omitted marker — and crawl runs under one wall-clock deadline with a UUID idempotency key that stays stable across HTTP retries. Ships as `firecrawl/firecrawl-php@1.9.0`, and the release pipeline now pushes proper `v{SDK_VERSION}` tags to firecrawl-php so Packagist finally exposes stable versions (`^1.9` resolves) instead of only `dev-main`.
- **Per-key concurrency limits on the FDB queue** — Added a third gate to the NuQ FoundationDB queue's concurrency chain — **crawl → key → team → ready** — so that jobs enqueued with an API key that has `api_keys.concurrency` set hold a per-key slot while queued/active on top of the existing team and crawl slots. Limits are read on the enqueue path through a 60 s Redis-cached read-replica lookup, apply automatically when every job in the batch (including crawl children, which inherit the kickoff key) shares one `apiKeyId`, and fail open — this is a throttle, not a security boundary. Keys without a limit are fully unaffected: no flag set, no key-gate keys touched, hot path unchanged. Requires the `ALTER TABLE api_keys ADD COLUMN concurrency integer;` migration.
- **`SELF_HOSTED_DOMAIN` env var** — Self-hosted instances behind a reverse proxy can now set `SELF_HOSTED_DOMAIN` to the externally-reachable host, fixing crawl status URLs, batch-scrape URLs, parse-upload links, and pagination `next` links that previously pointed at the internal bind address (e.g. `http://localhost:3002`). The resolver also honors `X-Forwarded-Host` and preserves ports; cloud behavior is unchanged when neither is set.
- **Express 5 upgrade** — Ported the API and worker to Express 5.2.1 with matching `@types/express` and `express-ws` updates, including explicit re-application of `express-ws` to the v1 and v2 routers so compiled routers expose `.ws` under Express 5.
- **TypeScript 7 stable** — Moved the API build compiler to `typescript@7.0.2` stable for roughly 3× faster clean builds (median ~1.3 s on `pnpm build` vs ~5.60 s on TS 5.9.2). The classic TypeScript programmatic API consumers (knip, `deterministicJson/pipeline/validate.ts`) resolve through a side-by-side `typescript → npm:@typescript/typescript6@6.0.2` alias, `pnpm knip` is now just `knip`, and all `tsc-watch` scripts and the harness `TscWatchClient` pass `--compiler ./node_modules/typescript-7/bin/tsc` since TS7's `exports` map doesn't publish `./bin/tsc`.
- **`threatProtection` option on JS & Python SDKs** — Exposed the v1/v2 `threatProtection` per-request override in `@mendable/firecrawl-js` (4.29.2 → 4.30.0) and `firecrawl-py` (4.31.0 → 4.32.0), matching the API's `threatProtectionOverrideSchema` exactly: `mode` (`off`/`normal`), `riskScoreThreshold` (0–100), `blacklist`, `whitelist`, `blockedTlds`, `failurePolicy` (`open`/`closed`). Accepted on scrape, batch scrape, crawl (via `scrapeOptions`), and top-level on search, map, extract, and agent in both SDKs; the Python `crawl` / `start_crawl` `threat_protection` kwarg is lifted into `scrapeOptions.threatProtection` on the wire; the JS SDK also omits `threatProtection` from `ParseOptions` so it can't leak into `/v2/parse` bodies.
- **`enterprise` search parameter across SDKs** — Wired the `/v2/search` `enterprise` option through the JS, Python, Ruby, and .NET SDKs so callers can select enterprise search modes (`["zdr"]` / `["anon"]`). Previously JS and Python silently dropped the field, and .NET typed it as `bool?` and serialized incorrectly. Ships as `@mendable/firecrawl-js@4.29.0`, `firecrawl@4.31.0` (Python), `firecrawl@1.10.0` (Ruby), and `firecrawl-sdk@1.9.0` (.NET). The Elixir SDK already wired this correctly.
- **PDF `totalPages` metadata** — PDF scrapes now expose a `metadata.totalPages` field carrying the document's true page count before any `maxPages` capping, so callers can tell a truncated response from a fully-parsed one. `totalPages > numPages` unambiguously signals truncation; the field is omitted when the true count is unknown (native detection failed), mirroring how `title` is handled. Threaded end-to-end through the PDF engine, v1/v2 response schemas, the GCS index cache, and the JS/Python SDK types.
- **Batched `/v2/search` highlights** — The `highlightsBeta` snippet path now sends every indexed web and news result to the model service in a single `/batch_highlight` request via a shared indexed-Chain helper, mapping responses back to results by stable page ID. Missing or empty highlights fall back to the original provider snippet (never a blank result), transient failures retry, and request IDs propagate through logging for cross-service tracing. A separate opt-in shadow path (`HIGHLIGHT_SHADOW_RATE`, `HIGHLIGHT_SHADOW_MAX_INFLIGHT`) can sample real traffic without mutating responses, drops overflow instead of queueing, and excludes ZDR requests.
- **Transparent gzip decompression** — Fire Engine scrapes now detect gzipped payloads by magic bytes and transparently decompress them via `gunzipSync` before parsing, with a 50 MB decompressed-size cap; size or decompression failures raise `UnsupportedFileError`. `application/gzip` is no longer excluded from specialty routing, so gzipped documents flow through the normal pipeline.
- **JS SDK `webhook` option on extract** — `startExtract` and `extract` in `@mendable/firecrawl-js` now accept `webhook?: string | WebhookConfig | null`, matching `crawl` and `batchScrape`. Callers can pass a URL string or a full webhook config with metadata and events, and it is forwarded to the request body.
- **`proxy: enhanced` / `stealth`** — Fixed `proxy: enhanced` and `proxy: stealth` silently falling back to non-stealth engines on the cloud — fallback selection now filters down to stealth-capable engines when stealth is requested, so the user's choice is actually honored. Self-hosted deployments without fire-engine keep working because the guard preserves the original engine list when no stealth-capable engine qualifies.
- **NuQ FoundationDB metrics** — Added FDB queue gauges alongside the existing NuQ metrics. `nuq_fdb_queue_<queue>_job_count` reports per-status counts (`queued`, `active`, `backlog`, `completed`, `failed`), `firecrawl_nuq_fdb_pending_jobs` exposes a worker-load gauge for autoscaling, and a new `GET /admin/<BULL_AUTH_KEY>/nuq-fdb-metrics` endpoint serves them in Prometheus format.
- **Autumn is the single source of truth for credits** — Removed the last ACUC-based credit accounting paths in the v0/v1/v2 hot path. The v0 `scrape`, `crawl`, and `search` controllers now check credits via `autumnService.checkCredits` (matching v1/v2), the unverified agent-key 50-credit cap reads `getTeamBalance().usage` from Autumn instead of `req.acuc.adjusted_credits_used`, and the dead `checkTeamCredits` / `supaCheckTeamCredits` / `evaluateTeamCredits` helpers have been removed. Behavior is preserved: 402 on insufficient credits, fail-open when Autumn is unavailable or self-hosted. v0 crawl no longer clamps `crawlerOptions.limit` to ACUC `remainingCredits` when Autumn allows the request (including overage), matching v1/v2. `fireclaw`'s controller-level multi-play credit check has also moved to Autumn `checkCredits` (`plays * 100`), with `remaining_credits` in the response now read from `getTeamBalance()`.
- **ACUC concurrency & rate-limit RPCs bumped to v55** — Pointed the API's `auth_credit_usage_chunk` RPCs at the latest `_55` chunk. That chain of migrations carries the v50 concurrency fixes — single-team orgs resolve concurrency as `GREATEST(override, plan)` (stops stale coupon overrides throttling paid teams), pay-as-you-go orgs get a concurrency floor of 5 instead of 2, and per-team `rate_limits` overrides are merged with the base plan per-endpoint via jsonb `||` so a partial override like `{"research": 500}` no longer reverts every other endpoint to free-tier limits — plus v51 surfacing `organization_data_source_terms` on the cached flags payload, v53's multi-team-org fallback that resolves plan-less teams in `can_have_multiple_teams` orgs to Scale/Enterprise limits + `plan_priority` (instead of free tier), and v52–v55 dropping the six zeroed credit columns (`price_credits`, `credits_used`, `coupon_credits`, `adjusted_credits_used`, `remaining_credits`, `total_credits_sum`), five deprecated subscription/price fields, and `sub_user_id` from the ACUC output — leaving Autumn as the sole source of truth for spend.

- Fixed deprecated v0 endpoints performing no threat protection checks — `/v0/scrape`, `/v0/crawl`, and `/v0/search` now reject `threatProtection: "forced"` teams with a 403 immediately after auth (mirroring the existing v0 forced-ZDR guard). `PUT /v2/team/threat-protection` also now rejects `mode: "off"` for forced teams, so the org policy can't be disabled with only an API key.

The cookie is now parsed and seeded into the browser context via `context.addCookies()`, scoped to the registrable domain, so authenticated pages that redirect before rendering no longer come back as the login page.
- Fixed `redactPII` and change-tracking diffs causing scrape-worker pods to shell out to `diff` for every page — the API now uses an in-process diff implementation, removing a fork-per-scrape and a hard dependency on `diff` being installed in the container.
- Fixed external `$ref` passthrough in `/v2/extract` schemas by replacing `json-schema-ref-parser` with an internal-only `$ref` resolver that refuses non-internal references, fixing a class of SSRF-shaped errors where extract requests with external `$ref` URLs could cause the worker to fetch arbitrary HTTP endpoints.
- Fixed v0 FoundationDB-backed crawl status reads and result blobs being routed to the wrong backend, so v0 crawl polling no longer returns stale or missing data on FDB clusters; FDB `waitForJob` now honors the caller-supplied timeout instead of clipping to the optional-mode wait.

- Fixed the FoundationDB native binding failing to build under pnpm by allowlisting the `foundationdb` package build script in both `apps/api/package.json` and `apps/api/pnpm-workspace.yaml`, restoring linux/arm64 image builds that include the native binary.

- Fixed self-hosted snip tests failing with `fetch failed` whenever `IDMUX_URL` was unreachable — self-hosted runs now fall back to the configured `TEST_API_KEY` / `TEST_TEAM_ID` with a warning, while production runs stay strict.

- Fixed the PHP SDK's `scrape`, `search`, and `map` methods returning empty results on API HTTP 200 responses with `success: false` (e.g. DNS failures upstream) — they now raise instead of silently hydrating empties, matching the Python and JS SDKs, and `CrawlOptions` gains an `idempotencyKey` field so retried crawl kickoffs no longer double-start.

## API

- Added `GET /v2/team/threat-protection` and `PUT /v2/team/threat-protection` for the flag-gated org threat-protection config (full-document update, zod-validated, audit-logged). Config is one row per org in `threat_protection_config`, keyed by `mode` (`off` / `normal`) plus a tolerantly-parsed `config` jsonb (`riskScoreThreshold`, `blacklist`, `whitelist`, `blockedTlds`, `failurePolicy`, `allowRequestOverrides`). Blocked requests raise `unsafe_domain_blocked` (403 on sync scrape/search/map/extract/agent; carried across queued crawl/batch jobs, `/errors`, and webhooks). Per-request `threatProtection` overrides mirror the org schema field-by-field on scrape, batch scrape, crawl (via `scrapeOptions`), search, map, extract, and agent (v1 + v2); requests that flip a `forced` policy to `mode: "off"` are rejected with 403. Billing: **+2 credits per unique scanned URL** per billing scope (crawl-wide dedup on discovered blocked links). Requires the `threat_protection_config` DDL.
- Added `POST /v2/scrape` (and every endpoint that shares its options) support for `formats: ["menu"]`, returning `document.menu` (canonical `MenuProfile` shape). Gated behind the `menuBeta` team flag and the `MENU_EXTRACTION_SERVICE_URL` self-host env var; teams without access get no `menu` field and no error.
- Added `POST /v2/parse/upload-url`, which mints a short-lived team-scoped `uploadRef` plus signed storage upload parameters. Authenticated only — keyless callers cannot mint upload refs. Refs are HMAC-signed and expire after 10 minutes, bound to the team, object path, filename, content type, and max size. New self-host env vars: `PARSE_UPLOAD_REF_SECRET` (required outside local/dev/test), `GCS_PARSE_UPLOAD_BUCKET_NAME`, and `PARSE_UPLOAD_STORAGE_DRIVER=gcs`.
- Added an `uploadRef` field to `POST /v2/parse` as an alternative to multipart uploads. The parse-by-ref request must authenticate as the same team that minted the ref; the existing multipart `/v2/parse` behavior is unchanged.
- Added a `metadata.totalPages` field to PDF scrape responses (v1/v2), carrying the document's true page count before any `maxPages` capping. `totalPages > numPages` signals truncation; the field is omitted when the true count is unknown. Threaded through the JS and Python SDK metadata types.
- Added a `search` monitor target on `POST /v2/monitor` and `PATCH /v2/monitor/:id` accepting `queries: string[]`, `searchWindow`, `includeDomains?: string[]`, `excludeDomains?: string[]`, and `maxResults`. `goal` is required only when `judgeEnabled` is true; partial updates validate the merged state.
